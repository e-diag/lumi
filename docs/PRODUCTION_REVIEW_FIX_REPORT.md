## 2. Найденные проблемы и что сделано

### 2.1 Утечка чувствительных данных в логах

- **Проблема**: `RequestLogger` писал полный `path` запроса → для `GET /sub/{token}` в JSON-логах оказывался **полный subscription token**.
- **Исправление**: пакет [`internal/logredact`](../internal/logredact/logredact.go), маскировка пути до `/sub/[redacted]`; в том же логе поле `remote` переведено на **`ClientIP(r)`** (согласовано с rate limit и учётом `RealIP`).

### 2.2 Идентификаторы ЮKassa в логах

- **Проблема**: при ошибке webhook и при «неизвестном» `yookassa_id` в лог уходил **полный** внешний ID.
- **Исправление**: в лог пишется сокращённая форма `ProviderPaymentIDForLog` (длина + суффикс), достаточная для грубой корреляции без полного значения.

### 2.3 Продакшен Docker и PostgreSQL

- **Проблема**: в [`docker-compose.prod.example.yml`](../docker-compose.prod.example.yml) порт **5432** публиковался наружу так же, как в dev-compose — для прода это лишний риск.
- **Исправление**: секция `ports` у сервиса `postgres` **убрана**; в комментарии описано, как при необходимости открыть только на `127.0.0.1`. В [`docker-compose.yml`](../docker-compose.yml) добавлено пояснение, что `5432:5432` — для локальной разработки.

### 2.4 Небезопасно большие выборки пользователей

- **Проблема**: рассылка в боте вызывала `List(..., 1, 100000)` — при росте базы это удар по памяти и БД.
- **Исправление**: в [`user_usecase`](../internal/usecase/user_usecase.go) введён потолок **500** записей на один запрос; рассылка в [`broadcast_handler.go`](../internal/handler/bot/broadcast_handler.go) итерирует **страницами по 500**.

### 2.5 Потолок размера страницы списка платежей

- **Проблема**: теоретически возможен вызов `ListByFilter` с огромным `pageSize` (если появится новый клиент).
- **Исправление**: жёсткий потолок **200** в [`payment_usecase.go`](../internal/usecase/payment_usecase.go), тест `TestPaymentUseCase_ListByFilter_CapsPageSize`.

### 2.6 Документация vs реальность

- **Проблема**: в [`docs/SETUP.md`](SETUP.md) webhook описывался расплывчато («подпись/сетевая политика»), при том что в коде реализован **только allowlist IP ЮKassa**.
- **Исправление**: уточнены формулировки в разделах про API и продакшен; в [`README.md`](../README.md) добавлена отсылка к prod-compose без публикации Postgres.

## 4. Что осталось legacy / ограниченно и почему

- **In-memory rate limit** для `/sub`: при очень большом числе уникальных IP карта растёт; при `> 200000` записей выполняется полный сброс карты (простая защита от бесконечного роста, не идеальная с точки зрения справедливости лимитов). Для высоконагруженного продакшена разумнее вынести лимиты в Redis/edge.
- **Ответ webhook 200 при внутренней ошибке** — сознательное решение не заставлять ЮKassa бесконечно ретраить; побочный эффект: при постоянном сбое активации нужен мониторинг логов/алертов и ручной или воркерный дожим (в SETUP уже есть подсказка про `payment_activations`).
- **Схема БД**: только AutoMigrate — для строгого enterprise-процесса обычно нужны версионируемые SQL-миграции и политика отката.

## 6. Что всё ещё не «идеальный прод»

- Нет **проверки подписи** тела webhook ЮKassa (полагаемся на IP + корректность JSON); при изменении модели угроз или сети ЮKassa — пересмотреть.
- **Горизонтальное масштабирование API**: воркеры в каждом инстансе — риск дублирования фоновых задач; нужен отдельный worker или leader election (уже отмечено в SETUP).
- **Рассылка в боте**: последовательная отправка всем пользователям может упираться в лимиты Telegram API; для больших баз нужны очереди/батчи/задержки.
- **E2E** оплаты и импорта подписки в клиентах на стенде не автоматизированы в репозитории.

## 8. Следующие шаги перед публичным запуском

1. Прогнать полный ручной сценарий: регистрация → триал → оплата → webhook → продление → импорт URL в Happ и v2RayTun на реальных приложениях.
2. Зафиксировать на сервере compose **без** публикации Postgres; TLS на reverse proxy; секреты вне git.
3. Настроить алерты по логам: `yookassa webhook failed`, `sub handler`, `payment: release activation after failure`.
4. При росте трафика: вынести rate limit на edge/Redis; при нескольких репликах API — вынести воркеры.
5. Запланировать переход с AutoMigrate на версионируемые миграции, если нужен формальный compliance.
